# Writing Plans

## Что делает

Помогает превращать идею, задачу или спецификацию в понятный пошаговый план:

- разбивает работу на атомарные шаги
- добавляет проверку выполнения на каждом шаге
- делает план пригодным для исполнения после паузы или рестарта

## Главный принцип

Пиши план так, будто его будет выполнять будущий-ты без контекста текущей сессии.

Никаких “как обсуждали выше” и “тот файл”. Только самодостаточные шаги.

## Шапка плана

```markdown
# [Название] - План реализации

**Цель:** Что должно получиться в итоге
**Срок:** Грубая оценка времени
**Зависимости:** Что нужно до старта
```

## Структура шага

Каждый шаг должен быть атомарным и проверяемым:

```markdown
### Шаг N: [Действие]

**Что делаем:** Конкретное действие
**Файлы:** Точные пути
**Команды:** Точные команды, если нужны
**Проверка:** Как проверить результат
**Результат:** Что получили после шага
```

## Правила хорошего плана

- Пиши точные пути, а не “тот конфиг”.
- Пиши точные команды, а не “запусти тест”.
- У каждого шага должна быть проверка.
- Не смешивай несколько независимых действий в один шаг.
- Не добавляй “на всякий случай” лишнюю работу.

## Гранулярность

### Технические задачи

- шаг = одно изменение, запуск или проверка
- указывай точные файлы и команды
- фиксируй ожидаемый результат

### Контентные задачи

- шаг = один этап: ресёрч, драфт, правка, публикация
- перечисляй источники, артефакты и критерии готовности

### Организационные задачи

- шаг = одно завершённое действие
- указывай адреса, контакты, дедлайны, если они есть

## После построения плана

Нормальные следующие варианты:

1. Выполнять план сразу.
2. Отдать выполнение отдельному агенту.
3. Сохранить план и вернуться позже.

## Признаки плохого плана

- “потом разберёмся”
- “если что, придумаем на месте”
- “исправить всё”
- отсутствие проверки после шага

Если это появляется, план надо переписать до внятного состояния.
